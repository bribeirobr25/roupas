# I18N.md — Internacionalização

> 4 idiomas: **EN, PT-BR, DE, ES**. Detecção automática por browser, fallback EN, seletor manual sempre disponível.

## 1. Detecção e fallback
- Ler `navigator.language` / `navigator.languages` no cliente.
- Mapear: `pt*` → PT-BR; `de*` → DE; `es*` → ES; `en*` → EN; qualquer outro → **EN** (fallback).
- Persistir a escolha manual (localStorage não é permitido em alguns ambientes de artifact, mas no app Vercel real é ok; usar cookie ou localStorage).
- Definir `<html lang="...">` conforme o idioma ativo (acessibilidade/SEO).
- Toda string da UI passa pelo i18n. **Nada hard-coded.** A API devolve `messageKey`/`noteKey` (camelCase), o frontend traduz.

## 2. Estrutura sugerida
Um objeto/arquivo por idioma com as mesmas chaves (`en.json`, `pt-BR.json`, `de.json`, `es.json`). Chaves abaixo. Valores extraídos de páginas (ex.: "100% organic cotton") **não** são traduzidos — são dados.

> **Implementação (2026-06-07):** dicionários em TS, não JSON (`lib/i18n/dictionaries/*.ts`), com `en` como **fonte de verdade tipada** (`Dict = typeof en`); os demais idiomas devem conformar à mesma forma (garantido por teste de paridade de chaves). Além do copy-base de §3, a implementação adiciona rótulos de UI: `app.headlineLead`/`app.headlineMain` (manchete), `input.analyzing`, `result.detectedCategory`/`categoryLow`/`quality`/`wrinkleQuestion`/`confidenceLabel`/`verifiedTag`/`inferredTag`, `category.*` (tshirt/shirt/pullover/hoodie/unknown), `finding.*` (composição/fibra/GSM/tecelagem/fiação/elastano/poliéster/non-iron/construção) e `value.*`. Nenhuma string hard-coded.
